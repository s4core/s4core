# Архитектурный план миграции: S4core (REDB ➔ FJALL)

Данный документ описывает фундаментальную трансформацию подсистемы метаданных и индексов проекта S4core. Переход с B-Tree (redb) на LSM-Tree (fjall) — это стратегический шаг для достижения колоссальной пропускной способности на запись сотен тысяч мелких объектов и подготовки ядра к распределенной (Distributed) среде в будущем.

Архитектура состоит из 8 последовательных фаз. Каждая фаза изолирует риски и гарантирует, что слой S3 API (Layer 1) и бизнес-логика (Layer 2) не пострадают в процессе замены фундаментального хранилища (Layer 3).

---

## Фаза 1: Проектирование абстракций (The Isolation Shield)
*Цель: Полностью отвязать текущую логику S4core от вендор-специфичных концепций B-Tree и ACID транзакций redb.*

1. **Создание `StorageEngine` Trait'а:**
   В модуле `s4-core/src/storage/engine.rs` необходимо объявить строгие абстракции, независящие от базы данных.
2. **Абстрагирование Batch-операций:**
   Вместо возврата транзакционных локов (Read/Write Transactions), API индекса должно принимать список мутаций (Mutations) и применять их атомарно через отдельный абстрактный метод `commit_batch(ops: Vec<IndexOperation>)`.
3. **Отказ от строгой типизации таблиц базы:**
   В LSM-деревьях используются пространства ключей (Keyspaces/Partitions) и байтовые массивы. Сериализация структур метаданных (`IndexRecord`, `BlobHeader`) полностью перекладывается на `bincode` до попадания байтов в абстракцию `IndexDb`.

## Фаза 2: Инициализация Fjall и Разметка Партиций (Data Layout)
*Цель: Внедрить зависимость fjall и правильно распределить логические сущности по LSM-деревьям во избежание I/O интерференции (Write Amplification).*

1. **Зависимости:** 
   Добавление `fjall` и `std::sync::Arc` в `s4-core/Cargo.toml`.
2. **Конфигурация Keyspaces Configuration:**
   Инициализация глобального `fjall::Config` в памяти и создание трех независимых партиций (`PartitionHandle`):
   - `partition_objects` — содержит маппинг `Bucket/Path -> IndexRecord`. Оптимизируется на чтение (Point-lookups).
   - `partition_dedup` — содержит маппинг `SHA-256 -> (File_Id, Offset, RefCount)`. Оптимизируется на атомарные инкременты и частую запись/уникализацию.
   - `partition_multipart` — временное стейт-хранилище незаконченных загрузок. Из-за частых удалений имеет агрессивные настройки фоновой компакции.

---

## Фаза 3: Адаптация Модели Данных и Интеграция Logical Clocks
*Цель: Подготовить структуры данных для High Availability, шардирования и ленивого разрешения конфликтов в распределенном будущем.*

1. **Внедрение Vector Clocks / Version IDs:**
   Реструктуризация `IndexRecord`. Добавление поля `logical_version: u64` (или `Lamport Clock`). Это критически важно для LSM-Tree (где данные не перезаписываются сразу, а добавляются новые версии) и для будущей Raft-репликации, чтобы узлы понимали кто "правее".
2. **Inline Storage для Tiny Objects (< 4KB):**
   Реализация прозрачной запаковки бинарных данных напрямую в `IndexRecord` без записи в append-only лог (Tombs/Volumes). Fjall, в отличие от redb, работает в RAM через Block Cache, поэтому чтение таких объектов будет происходить со скоростью оперативной памяти (Zero-copy read access).

---

## Фаза 4: Реализация Ядра Индекса (The Core Write/Read Paths)
*Цель: Написать базовые операции CRUD для `s4-core/src/storage/index.rs` поверх `fjall`.*

1. **Оптимистичное Чтение (Point-Lookups):**
   Использование встроенных в fjall Bloom Filters для мгновенного определения отсутствия ключа. Это исключает ненужные походы к диску при запросе несуществующих файлов (404 Not Found).
2. **Write-Ahead Log (WAL) & Батчинг:**
   Переписываем `write_metadata`. Создаем `fjall::Batch`, добавляем туда `IndexRecord`, и, если объект новый, обновляем счетчик в `partition_dedup`. Если тома (Volume) успешно синхронизировал файл (fsync), коммитим весь LSM-батч одним вызовом. Интеграция гарантирует консистентность метаданных и бинарного тома без потерь при `kill -9`.

---

## Фаза 5: Атомарность Директорий и Механизмы Дедупликации
*Цель: Сохранить "киллер-фичи" проекта и сделать их безопасно-конкурентными.*

1. **Виртуальные Директории (Атомарное Переименование):**
   Реализация Prefix Scan через итератор `fjall` по `partition_objects`. 
   Для переименования папки:
   - Ищем все ключи, начинающиеся с `shard_id/old_dir/`.
   - Начинаем собирать `fjall::Batch`.
   - В итераторе делаем `batch.insert(new_key, value)` и `batch.remove(old_key)`.
   - Применяем `db.commit(batch)`. Транзакция сверхбыстрая, так как пишется только в LSM Memtable (ОЗУ).
2. **Дедупликация и Сборка Мусора (Garbage Collection):**
   Из-за отсутствия глобального лока (`db.write()` в redb), вводим In-Memory Mutex/RwLock шардированный по хешам (шлюз) поверх слоя дедупликатора для single-node, что предотвращает состояния гонки при конкурентной заливке одинаковых бинарников.

---

## Фаза 6: Background Compaction (Фоновая Очистка и Tombstones)
*Цель: Настроить правильное сжатие устаревших данных и синхронизировать его с ротацией лог-файлов S4core.*

1. **Фоновая компакция LSM (Level/Tiered):**
   Настройка фоновых тредов fjall. Удаленные объекты (DELETE Object) просто получают "Tombstone" метку в fjall.
2. **Garbage Collection Link (Интеграция с Volume-файлами):**
   Написание сервиса Scrubber, который сканирует старые append-only тома. Если `IndexDb` говорит, что блок мертв (так как в LSM лежит Tombstone маркера файла, либо `ref_count` в `partition_dedup` == 0), фоновый воркер игнорирует блок при переупаковке `volume_XXX.dat` в новый том. Никакого блокирования входящих HTTP-запросов.

---

## Фаза 7: Тестирование, Бенчмарки и Инструменты Миграции
*Цель: Обеспечить безопасный переезд с B-Tree, не потеряв ни один байт.*

1. **Creation of `s4-cli migrate` tool:**
   Утилита для бесшовной миграции: открывает на чтение старый файл `redb`, итерируется по всем записям и перекладывает их в `fjall` используя параллельные батчи.
2. **Стресс-тестирование High Ingest Rate:**
   Бенчмарк-тестирование на 1-10 миллионах объектов (< 4KB). Ожидается рост пропускной способности при записи минимум на 40-70% из-за природы LSM (отсутствие random write I/O на уровне диска, которые присущи B-Tree при ребалансировках узлов).

---

## Фаза 8: Подготовка State Machine (Distributed Readiness)
*Цель: Сделать движок готовым к интеграции с Raft или другим алгоритмом консенсуса, чтобы S4core стал распределенным по щелчку пальцев.*

1. **State Machine Interface:**
   Модификация `StorageEngine` для внедрения контракта `apply(&self, event: LogEntry)`. 
   Любое изменение в S4core теперь является детерминированным (лог репликации).
2. **Local-to-Shard Deduplication:**
   Проектирование логики, где Hash-роутер (снаружи) отправляет объект в правильную Ноду. "Умная" (распределенная) дедупликация теперь локализована. Модуль `s4-core` дедуплицирует данные только в пределах своей физической ноды (через `partition_dedup`), что полностью снимает проблему глобальных распределенных блокировок координатора при CAS (Content Addressable Storage).
